# Frontend / UI AI Handoff Brief

更新时间：2026-04-12

适用对象：

- Gemini
- Claude
- GPT
- 其他负责前端、交互、视觉重构的 AI 模型

适用范围：

- 仅限本项目的前端布局、交互体验、信息架构、阅读体验、视觉层级优化
- 不包含后端重写
- 不包含支付、鉴权、任务链路的业务改造

---

## 1. 项目一句话说明

这是一个面向短剧生产前置环节的内容工作台，核心链路为：

`原文 -> 分析 -> 大纲 -> 剧本 -> 分镜 -> 导出`

它不是营销官网，也不是单页生成器，而是一个正在持续迭代中的生产型 SaaS 工作台。

---

## 2. 当前项目真实边界

### 2.1 当前已稳定的核心能力

- 项目工作台已可用
- 原文上传、编辑、保存可用
- 生成链路可跑通：
  - 分析
  - 大纲
  - 剧本
  - 分镜
  - 导出
- Google-only 登录已修复并稳定
- PayPal live 支付链路已修复并实测通过
- 中文界面正在持续补齐
- LLM provider 已统一为可兼容 OpenAI 协议的网关方式
- 分镜阶段已有多层 fallback，稳定性优先

### 2.2 当前不是本轮要做的事

- 不接视频生成 API
- 不做图片素材管理系统
- 不做新的支付体系
- 不重做认证体系
- 不重写 Job / Artifact / Workspace 架构
- 不推翻现有原文到分镜的生产链路

---

## 3. 当前前端的主要问题

从产品和使用体验看，当前前端的核心问题不是“功能没有”，而是“表达方式不够清晰”。

### 3.1 信息结构问题

- 页面更像“工具集合”，而不是“创作工作流”
- 主流程阶段感不够强
- 原文、剧本、分镜之间的层级关系不够直观
- 预览区和操作区的角色混在一起

### 3.2 阅读体验问题

- 生成后的文本浏览体验较差
- 长文本、剧本、分镜的可读性不够
- 信息块之间节奏不稳定
- 用户需要花较多认知成本理解“现在在哪一步”

### 3.3 交互反馈问题

- 阶段切换不够“工作流化”
- 下一步动作不够明确
- 状态反馈与内容区没有形成强关联
- 成功、失败、处理中虽然存在，但视觉分层还不够强

### 3.4 UI 方向问题

- 页面结构偏“后台工具”，不是“内容工作台”
- 缺少统一的主视觉秩序
- 需要更像“创作台 / 文档工作区 / 流程驾驶舱”

---

## 4. 本轮 UI / 交互改造目标

本轮改造不是换皮，而是重新组织工作台的表达方式。

### 4.1 结构目标

把顶部主流程明确组织为三大块：

- 小说
- 剧本
- 分镜

要求：

- 三大块应作为主导航和主状态区
- 用户一眼就能看懂当前位于哪一阶段
- 每个阶段应该有自己的状态、进度、摘要和入口

### 4.2 内容目标

中下部区域应更像“内容工作区”而不是单纯的工具面板集合：

- 当前阶段说明
- 主预览区
- 次级操作区
- 辅助状态区

### 4.3 阅读目标

提升以下内容的阅读体验：

- 原文
- 分析稿
- 大纲
- 剧本
- 分镜结构

重点不是增加更多组件，而是：

- 提升节奏
- 控制密度
- 强化重点
- 降低阅读疲劳

### 4.4 交互目标

- 强化“当前阶段”
- 强化“下一步”
- 强化“我现在该做什么”
- 强化任务状态和内容预览之间的关系

---

## 5. 严格红线

以下内容不允许被外部 AI 在 UI 改造过程中破坏。

### 5.1 不允许破坏业务链路

- 不改 API 协议
- 不改后端 route 输入输出语义
- 不改任务流转顺序
- 不改 Artifact 基本类型和关系
- 不删除当前可用的 fallback 机制

### 5.2 不允许破坏登录与支付

- 不破坏 Google 登录能力
- 不把登录体系改回邮箱密码杂糅方案
- 不破坏 Google-only 登录策略
- 不改坏 Clerk session bridge
- 不破坏 PayPal live 支付
- 不回退 Creator 套餐价格显示

### 5.3 不允许破坏国际化

- 中文界面必须完整中文
- 不允许新增明显英文漏出
- 不允许只做英文文案后再“顺手翻译”
- 必须确保中文信息密度和中文排版合理

### 5.4 不允许推翻现有架构

- 不允许为了“代码更优雅”而整体重写工作台
- 不允许删除现有工作组件再重新发明一套状态系统
- 不允许擅自替换现有页面的数据来源

---

## 6. 允许改动的范围

以下内容允许外部 AI 调整：

- 页面信息架构
- 区块层级
- 组件排布
- 样式系统
- 字体层级
- 卡片关系
- 标签、说明文案、按钮文案
- 状态展示方式
- 阶段导航方式
- 阅读区 / 预览区布局
- 空状态、加载态、错误态的视觉组织
- 动效节奏
- 移动端适配

只要不触碰上面的红线，这些都可以改。

---

## 7. 推荐的设计方向

### 7.1 页面气质

页面应更接近：

- 创作工作台
- 文档工作区
- 流程驾驶舱

不应继续停留在：

- 普通后台表单页
- 组件堆砌页
- 杂乱工具台

### 7.2 建议结构

推荐遵循以下结构：

1. 顶部：项目标题 + 全局状态
2. 主流程区：小说 / 剧本 / 分镜 三大块
3. 当前阶段区：说明、状态、子导航
4. 主预览区：文档/剧本/分镜主内容
5. 侧边辅助区：操作项、版本、过滤、任务提示
6. 底部动作区：下一步 / 导出 / 继续生成

### 7.3 视觉建议

- 用更明确的块级关系表达阶段差异
- 阶段区可以带弱色彩区分，但不要过度花哨
- 预览区要更安静、更适合阅读
- 操作区可以更紧凑，预览区应获得更多空间
- 中文字号、行高、段落间距要重新校准

---

## 8. 交互原则

### 8.1 先看懂，再操作

用户进入页面后，应优先理解：

- 项目是什么
- 当前进度到哪
- 当前页属于哪个阶段
- 下一步该做什么

### 8.2 内容优先于控件

这个页面的核心是内容生产，不是按钮集合。

因此应遵循：

- 先内容
- 后工具
- 主预览清晰
- 次操作收敛

### 8.3 阶段感必须明显

从任意页面进入工作台，用户都应快速感知：

- 小说阶段
- 剧本阶段
- 分镜阶段

而不是先去理解内部的很多 tab。

### 8.4 不新增无意义复杂度

不要为了显得“高级”而引入：

- 过多悬浮交互
- 复杂手风琴嵌套
- 过量视觉噪音
- 难以维护的奇怪布局

---

## 9. 性能与工程约束

外部 AI 在修改 UI 时，需要同时满足以下工程约束：

- 页面交互不能更慢
- 不得引入明显的客户端卡顿
- 不得造成按钮点击无反馈
- 不得让工作台首屏层级更重
- 不得破坏现有轮询刷新逻辑
- 不得在未必要时引入新的全局状态管理

应优先复用现有状态和数据，而不是重新造状态。

## 11. 外部 AI 的推荐工作方式

建议要求外部 AI 严格按下面顺序工作。

### 第一步：先理解，不要直接写代码

先输出：

- 当前页面结构理解
- 当前主要问题判断
- 推荐改造方向
- 哪些文件会改
- 哪些内容不会动

### 第二步：先给低风险重构方案

优先做：

- 布局重组
- 信息架构优化
- 阶段导航优化
- 阅读体验优化

不要第一步就：

- 大量抽象组件
- 全量重命名
- 改数据流
- 改接口

### 第三步：逐步提交

建议分成三批：

1. 工作台结构层改造
2. 预览与阅读层改造
3. 交互细节与移动端适配

### 第四步：每次提交都带验证说明

必须说明：

- 改了哪些文件
- 为什么这样改
- 是否触碰接口和业务逻辑
- 怎么验证没破坏主链路

---

## 12. 建议外部 AI 的交付物

要求它最终给出这些内容：

### 12.1 设计方案摘要

- 现状问题
- 改造方向
- 主要变化点

### 12.2 实际代码改动

- 改动文件列表
- 每个文件的改动目的

### 12.3 风险说明

- 哪些地方是纯视觉
- 哪些地方影响交互
- 哪些地方可能影响已有组件行为

### 12.4 验证清单

至少覆盖：

- 原文编辑
- 原文上传
- 阶段切换
- 剧本预览
- 分镜预览
- 导出入口
- 任务状态查看
- 中文显示完整性
- 移动端布局

---

## 13. 最终验收标准

### 13.1 结构验收

- 顶部必须明确体现三大阶段：小说 / 剧本 / 分镜
- 页面结构明显比现在更清晰
- 用户能快速知道当前处于哪个阶段

### 13.2 阅读验收

- 长文本更易读
- 剧本更易扫读
- 分镜信息更易检视
- 视觉噪音减少

### 13.3 交互验收

- 切换有清晰反馈
- 下一步动作明确
- 状态与内容形成关联

### 13.4 工程验收

- `npm run build` 通过
- 不破坏主功能链路
- 不新增明显英文漏出
- 不让页面交互变慢

## 15. 如果使用 Gemini，需要特别注意的事

### 15.1 不要把密钥直接写进提示词

不要把真实的 API key 明文贴给模型。

### 15.2 Gemini 更适合做这些事

- 输出页面重组方案
- 给出组件级改造建议
- 做局部前端重构
- 做视觉文案和交互方案

### 15.3 不建议直接让它做这些事

- 全仓库无约束重写
- 擅自重构业务链路
- 修改认证与支付逻辑
- 大范围“顺手清理”项目

---

## 16. 一句话总结

交给其他 AI 的正确方式不是：

“帮我把页面改漂亮。”

而应该是：

“请在不破坏生产链路的前提下，把这个内容工作台重组为更清晰、更适合阅读和创作的三阶段工作流界面。”
